# Stellar Mixer RISC Zero Prover

This repository contains the RISC Zero guest programs used by Stellar Mixer.

Its main purpose is not to be a production server.

Its main purpose is **verification**.

Anyone should be able to:

    - read the Rust source code of the mixer guest programs
    - compile those guest programs locally
    - export the resulting RISC Zero image IDs
    - compare them with the image IDs used by the Stellar Mixer contract

This allows users, auditors, and independent operators to verify what code the mixer contract accepts proofs for.

No trusted party is needed for this step.

## Why this repository exists

The Stellar Mixer contract verifies zero-knowledge proofs against fixed RISC Zero image IDs.

Those image IDs could simply be written in documentation, but that would not be enough.

A user should not have to trust that a published image ID corresponds to a specific Rust guest program.

This repository provides the missing link:

    guest Rust source code
        -> compiled RISC Zero guest ELF
        -> RISC Zero image ID
        -> image ID accepted by the mixer contract

If a user compiles this repository and gets the same image IDs as the deployed contract, they can verify that the contract is checking proofs for these guest programs.

That is the point of this repo.

## Trust model

The user does not need to trust:

    - the project website
    - the desktop client
    - the Groth16 wrapper server
    - an off-chain prover
    - a public API server
    - a README statement saying "this is the correct image ID"

Instead, the user can verify the source-to-image-ID relationship directly.

The trust path is:

    1. Inspect the guest Rust code.
    2. Build the guest programs.
    3. Export the image IDs.
    4. Compare them with the image IDs configured in the mixer contract.
    5. Accept only if they match.

If the image IDs match, then the user knows which guest programs the contract accepts.

## What is an image ID?

In RISC Zero, an image ID identifies a specific guest program.

For Stellar Mixer, this matters because the contract should only accept proofs generated by the intended mixer programs.

Changing guest code changes the compiled guest ELF.

Changing the guest ELF changes the image ID.

That means the image ID is the compact identifier of the ZK program being proven.

## Repository structure

    Cargo.toml
      Workspace definition.

    rust-toolchain.toml
      Pinned Rust toolchain used by this repository.

    Cargo.lock
      Locked dependency graph. This must be committed and used with --locked.

    methods/
      RISC Zero method crate. It embeds the guest programs and exposes
      generated ELF and image ID constants.

    methods/transfer_guest/
      Guest program for transfer proofs.

    methods/withdraw_guest/
      Guest program for withdraw proofs.

    host/
      Host-side examples and helper binaries.

    host/src/main.rs
      Withdraw STARK proving benchmark.

    host/src/bin/export_artifacts.rs
      Exports guest ELF files and image IDs into artifacts/.

## Guest programs

There are two mixer guest programs.

### Transfer guest

The transfer guest proves that a user can consume existing notes and create new output notes while preserving value.

It checks:

    - input note membership against a Merkle root
    - ownership through RSA signature verification
    - input value equals output value
    - nullifier uniqueness inside the proof
    - output leaf commitment construction

It commits public transfer data into the journal:

    - Merkle root
    - nullifiers
    - output leaves

### Withdraw guest

The withdraw guest proves that a user can consume existing notes, withdraw a public amount, and create a change note.

It checks:

    - input note membership against a Merkle root
    - ownership through RSA signature verification
    - input value equals withdraw amount plus change value
    - nullifier uniqueness inside the proof
    - change note belongs to the same owner
    - change note does not reuse an input nullifier

It commits public withdraw data into the journal:

    - Merkle root
    - nullifiers
    - withdraw amount
    - output leaf for change

## Requirements

This repository builds RISC Zero guest programs.

That is different from building a normal Rust CLI program.

You need:

    - Rust / rustup
    - Cargo
    - Git
    - RISC Zero toolchain installed through rzup
    - normal native build tools for your OS

Recommended platforms:

    - macOS
    - Linux
    - WSL on Windows

Native Windows may work, but WSL/Linux is recommended for fewer build environment surprises.

## Install Rust

Install Rust using rustup if it is not already installed.

Then enter the repository directory.

The pinned toolchain from rust-toolchain.toml should be selected automatically by rustup.

Check:

    rustc -Vv
    cargo -V

This repository currently pins:

    rustc 1.96.0

## Install RISC Zero tools

Install the RISC Zero toolchain:

    curl -L https://risczero.com/install | bash
    rzup install

Then verify that the RISC Zero toolchain is available:

    rustup toolchain list --verbose | grep risc0

If your shell cannot find `rzup` after installation, restart the terminal or reload your shell profile.

## Native build dependencies

On macOS, install Xcode command line tools:

    xcode-select --install

On Ubuntu/Debian, install common build tools:

    sudo apt-get update
    sudo apt-get install -y build-essential clang cmake pkg-config git curl

Depending on your environment, additional native packages may be required by transitive dependencies.

If Cargo reports a missing compiler, linker, CMake, pkg-config, or system library, install the missing native package and rerun the same locked command.

## Build

Use locked builds.

Do not use plain `cargo build` for reproducibility checks.

Run:

    cargo build --locked

The `--locked` flag is important. It tells Cargo to use the committed Cargo.lock dependency graph and fail instead of silently changing dependency resolution.

## Export image IDs and ELF files

Run:

    cargo run --locked -p stellar-mixer-risc0-prover-host --bin export_artifacts

This writes files into:

    artifacts/

Generated files:

    artifacts/transfer_guest.elf
    artifacts/transfer_guest_image_id.bin
    artifacts/transfer_guest_image_id_hex_le_words.txt
    artifacts/transfer_guest_image_id_words.txt

    artifacts/withdraw_guest.elf
    artifacts/withdraw_guest_image_id.bin
    artifacts/withdraw_guest_image_id_hex_le_words.txt
    artifacts/withdraw_guest_image_id_words.txt

Print the generated image IDs:

    cat artifacts/transfer_guest_image_id_hex_le_words.txt
    cat artifacts/withdraw_guest_image_id_hex_le_words.txt

Expected values:

    df81b36b05b68434b9f3c78ab3eb10d12eee3fb9bdda54d77681f7055a2f0b44
    1d48517b74b17df6edfc93591254f8a9194023018753e74cb397a172739d0a86

If the generated values match the contract values, the local build reproduced the guest image IDs.

## Run withdraw STARK benchmark

The host also contains a simple withdraw STARK proving benchmark.

Run:

    cargo run --locked -p stellar-mixer-risc0-prover-host --bin stellar-mixer-risc0-prover-host

This example:

    - creates demo notes
    - builds a demo sparse Merkle tree
    - creates a withdraw STARK receipt
    - verifies the STARK receipt locally
    - prints proving time
    - prints journal data

It does not perform Groth16 wrapping.

It does not send anything to a wrapper server.

It is only a local benchmark/example for the withdraw guest.

## Relationship to Groth16 wrapping

This repository defines the RISC Zero guest programs and their image IDs.

Groth16 wrapping is a separate step.

The Groth16 wrapper server takes an already-created RISC Zero receipt and compresses it into a Groth16 receipt.

The wrapper server does not define mixer semantics.

The important trust boundary is:

    guest source code
      -> image ID

The contract verifies proofs against image IDs.

The wrapper server is only compute infrastructure.

## Relationship to the Stellar Mixer contract

The contract should accept proofs only for known image IDs.

A user can audit the contract by checking which image IDs it accepts.

Then the user can audit this repository by rebuilding the guest programs and verifying that the resulting image IDs match.

That gives the full verification chain:

    source code in this repository
      -> local RISC Zero guest build
      -> local image IDs
      -> deployed contract image IDs

No off-chain server has to be trusted for this relationship.

## Reproducibility notes

This repository pins:

    - Rust toolchain in rust-toolchain.toml
    - direct dependency versions in Cargo.toml
    - git dependencies by revision
    - full dependency graph in Cargo.lock

For reproducibility checks, always use:

    cargo build --locked
    cargo run --locked -p stellar-mixer-risc0-prover-host --bin export_artifacts

The image IDs should be reproducible across normal macOS/Linux/WSL environments when the same locked build inputs are used.

If a different OS or environment produces different image IDs, treat that as a failed reproducibility check and investigate before trusting the result.

## What changes image IDs?

Any change to the guest program can change the image ID.

This includes changes under:

    methods/transfer_guest/
    methods/withdraw_guest/

It can also include changes to:

    - RISC Zero version
    - guest dependencies
    - Rust toolchain
    - build configuration
    - Cargo.lock
    - method build configuration

Host-side benchmark code does not define mixer semantics by itself.

But if a host dependency change also changes the shared workspace lockfile or method build inputs, image IDs should be rechecked.

## Maintainer checklist

Before publishing changes:

    cargo build --locked
    cargo run --locked -p stellar-mixer-risc0-prover-host --bin export_artifacts
    cat artifacts/transfer_guest_image_id_hex_le_words.txt
    cat artifacts/withdraw_guest_image_id_hex_le_words.txt

Verify that the printed image IDs match the image IDs expected by the deployed contract.

Do not update the documented image IDs unless the mixer contract is intentionally upgraded or redeployed to accept new ones.

## Summary

This repository lets anyone verify the RISC Zero code behind Stellar Mixer proofs.

The image IDs are not just magic constants.

They are reproducible outputs of the guest Rust programs in this repository.

A user can build the guests locally, export the image IDs, and compare them with the mixer contract.

That is the reason this repository exists.
